# Project Report: ContractSimplifier

**Live URL**: http://3.213.82.133

## Tech Stack
- **Frontend**: React, Vite (Client-side rendering, dynamic UI)
- **Backend**: Python, FastAPI (High-performance async API)
- **AI / LLM Integration**: Groq API (Primary - `llama-3.3-70b-versatile`), Google Gemini API (Fallback - `gemini-2.5-flash`)
- **Deployment**: Docker (Multi-stage builds), AWS EC2 (Ubuntu Linux VPS)

## Prompting Strategy & Sample Prompts
Our strategy involves strict role-playing and structured JSON output to ensure the frontend can parse and render the results predictably. The prompt forces the LLM to act as an objective legal analyst.

**Sample System Prompt**:
```text
You are an expert legal assistant. Your task is to analyze a given contract clause and provide a plain-English explanation, assign a risk level (LOW, MEDIUM, HIGH), and provide a brief reasoning for the risk level.

You must output STRICTLY in JSON format matching this schema:
{
  "explanation": "Simple translation of the clause",
  "risk_level": "LOW|MEDIUM|HIGH",
  "reasoning": "Why this risk level was chosen"
}
```

## Phase-by-Phase Summary
- **Phase 1 (Backend Core)**: Set up the FastAPI structure, including route definitions, Pydantic models for request validation, and the foundational logic for extracting text from PDFs and raw text inputs.
- **Phase 2 (LLM Integration)**: Integrated the Groq API for rapid clause analysis. Implemented Google Gemini API as an automatic fallback mechanism to handle potential rate limits. Configured Server-Sent Events (SSE) to stream analysis results chunk-by-chunk to the client.
- **Phase 3 (Frontend UI)**: Developed the React-based frontend using Vite. Created a user-friendly interface for document uploads and text pasting, alongside a dynamic, real-time result viewer that processes the incoming SSE stream to display color-coded risk levels.
- **Phase 4-6 (Deployment & Documentation)**: Containerized the entire application (React + FastAPI) into a single Docker image using a multi-stage build. Deployed the container onto an AWS EC2 instance. Handled networking, security groups, and documentation for replication.

## Challenges + Fixes
1. **Challenge**: Delivering fast user feedback despite LLM latency on large contracts.
   **Fix**: We implemented Server-Sent Events (SSE) in FastAPI. Instead of waiting for the entire contract to be processed, we stream individual clause analyses as soon as they are generated by the LLM.
2. **Challenge**: Relying on a single free-tier LLM provider (Groq) often leads to rate-limiting errors.
   **Fix**: We built a robust fallback client. If the Groq API fails or hits a rate limit, the backend automatically catches the exception and routes the request to the Google Gemini API.
3. **Challenge**: Deploying a modern split-stack (React + FastAPI) typically requires managing two separate servers or containers.
   **Fix**: We used a multi-stage Dockerfile to build the React static files in an alpine-node environment, and then copied them into the Python container, where FastAPI mounts and serves them using `StaticFiles`. This resulted in a lightweight, easily deployable single container.

## Reflection
Building ContractSimplifier proved to be an excellent exercise in full-stack AI development. We successfully bridged the gap between complex legal language and everyday users by leveraging modern LLMs. The project highlighted the importance of robust error handling (like API fallbacks) and the user experience benefits of streaming data (SSE). Overall, deploying a fully containerized app on AWS solidified our understanding of production-ready architectures and cloud deployments.
